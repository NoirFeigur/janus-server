## 〇、系统全景（30 秒回顾，细节见架构稿）

公司统一内部 AI 基础设施平台，一个模块化单体（FastAPI），纳管两类 AI 能力：

```
客户端（Hermes / Claude Code / OA / 脚本）
   │ 同一套身份：企微 JWT  或  sk-xxxx
   ▼
统一平台（src/ 一个 repo）
   ├─ auth/     认证 + RBAC + 数据权限           （地基，M1）
   ├─ admin/    管理后台 API（用户/角色/部门/号池/模型/配额）  （M1 部分 + M3）
   ├─ gateway/  LLM 网关：三协议入口 + litellm.Router  （M2，未动工）
   ├─ mcp/      标准 MCP 服务器（FastMCP，零新表）        （M4，未动工）
   └─ tasks/    ARQ 后台任务（用量汇总/配额重置/清理）     （M5，骨架存在）
   共享：PostgreSQL + Redis
```

身份模型（架构稿 G5）：**主体落库统一是 `sys_user.id`（代码/API 命名 `user_id`）**；凭据分层：管理后台与资源管理 API 只接受平台 JWT/企微 JWT；LLM 推理协议端点与 MCP 协议端点额外接受 `sk-key` 作为程序调用凭据。二者最终都归因到同一个 `user_id` 与配额桶，但 `sk-key` 不是后台登录态。

- **M1 先于一切业务面**：网关与后台都依赖鉴权地基；先做地基，否则各自重造一套鉴权。

- **M3 依赖 M1 的 RBAC**：后台 CRUD 的权限门禁必须建立在 M1 判定之上。
- **M4 晚于 M2/M3**：MCP 经服务层暴露能力，须等服务边界稳定。
- **M6 最后做**：企微是外部适配器，用户/RBAC 用平台自签凭证已可独立闭环、完全可测。

## 三、数据模型现状与漂移（对齐数据模型设计 v0.5.5）

### 3.1 已落地 16 张表

`src/db/models/` 按领域拆分，全部继承三类基类之一：

| 文件 | 模型（类名 → 表名） |
|---|---|
| `identity.py` | User→`sys_user`、Department→`sys_department`、UserOAuth→`sys_user_oauth`、Role→`sys_role`、Menu→`sys_menu`、UserRole→`sys_user_role`、RoleMenu→`sys_role_menu`、RoleDept→`sys_role_dept` |
| `credential.py` | ApiKey→`api_key` |
| `model_catalog.py` | UpstreamChannel→`upstream_channel`、ChannelKey→`channel_key`、LogicalModel→`logical_model`、ModelDeployment→`model_deployment` |
| `grant.py` | UserModelGrant→`user_model_grant` |
| `usage.py` | UsageRecord→`usage_record`（`LogEntity`） |
| `quota.py` | Quota→`quota` |

三类基类（[`src/db/base.py`](../src/db/base.py)）：
- `BaseEntity`：`id / is_deleted / created_by / create_dept / created_at / updated_by / updated_at`（12 张配置/业务表继承）
- `LogEntity`：`id / created_at`（append-only 流水，UsageRecord）
- `LinkEntity`：`id / created_at`（关联表，物理删）

### 3.2 ✅ `BaseEntity.create_dept` 已补齐（部门级数据权限硬前提）

**现状**：`BaseEntity` 已新增 `create_dept`（普通索引），首迁移同步覆盖所有配置/业务表。`UserService`、`RoleService`、`DepartmentService` 创建数据时从当前用户部门写入。

**语义决策**：
1. user 域仍按 `User.department_id` 过滤，因为用户管理屏关注「目标用户属于哪个部门」。
2. role 域与后续通用业务表按 `create_dept` + `created_by` 过滤，因为它们是管理资源，部门归属应取创建者当时所在部门。
3. `LogEntity` 与 `LinkEntity` 不加 `create_dept`：流水 append-only，关联表是纯链接。

### 3.3 ✅ 表名边界：identity/RBAC 用 `sys_`，业务表不用

当前代码与数据模型文档一致：身份/RBAC 基础表使用 `sys_` 前缀（`sys_user` / `sys_role` 等），AI 网关业务表保持短表名（`api_key` / `usage_record` / `quota` / `upstream_channel` 等）。

## 四、鉴权与权限体系现状（全链路核实）

> 这是用户重点追问的部分：登录接口、统一拦截、开放白名单、接口权限、数据权限、超管 bypass。逐条给代码现状 + 缺口。

### 4.1 登录接口

**✅ 已落地**（[`src/auth/router.py`](../src/auth/router.py)）：

| 端点 | 方法 | 说明 |
|---|---|---|
| `/auth/login` | POST | 账密 → RS256 自签 access token |
| `/auth/me` | GET | 当前用户 + 权限码集合（需 JWT；返回 `user_id`） |

**服务层解析器**（无 HTTP 面，[`src/auth/service.py`](../src/auth/service.py)）：
`resolve_access_token`（JWT 验签）、`resolve_api_key`（sk-key → sha256 → 查 `api_key`）、`authenticate_password`（发 JWT）。

**⚠️ 缺口**：refresh（恒 None，留 M6）、logout（无吊销机制）、改密、个人中心、企微回调（`OAuthSource.wecom` 枚举 + `UserOAuth` 模型存在，但无端点无逻辑，留 M6）。

### 4.2 ✅ 统一登录拦截：已落地

[`src/main.py`](../src/main.py) 已挂 `AuthMiddleware`（位于 trace-id 之后的请求链路内，401 信封带 trace_id）。
中间件对所有非公开路径默认要求有效凭据，解析后写 `request.state.user`。

凭据边界：
- 管理后台、`/auth/me`、资源管理类 CRUD：只接受 JWT，走 `get_current_jwt_user` / `CurrentJwtUser`。
- LLM 推理协议端点：允许 JWT 或 sk-key，当前白名单为 `POST /v1/chat/completions`、`POST /v1/messages`、`POST /v1beta/models/{model}:generateContent`、`POST /v1beta/models/{model}:streamGenerateContent`。
- MCP 协议端点：允许 JWT 或 sk-key，当前只放开单端点 `/mcp`（GET/POST）；不放开 `/mcp/*` 资源/管理子路径。
- RBAC 仍由 `RequiredPerms` 处理，统一拦截只负责“有没有合法用户”和“凭据类型是否允许”。

### 4.3 ✅ 开放接口白名单：已落地

[`src/auth/middleware.py`](../src/auth/middleware.py) 定义中央 `PUBLIC_PATHS`：
`/auth/login`、`/health/live`、`/health/ready`、`/docs`、`/docs/oauth2-redirect`、`/openapi.json`、`/redoc`。
白名单支持 `api_prefix` 归一化，避免部署到 `/api` 后误拦截。

### 4.4 ✅ 接口权限（perms 码）：admin 域完善

`RequiredPerms("域:资源:动作")` 依赖工厂（[`src/auth/dependencies.py`](../src/auth/dependencies.py)），挂在全部 12 个 admin 端点：

```
users:  system:user:{list,add,edit,remove}
roles:  system:role:{list,add,edit,remove}
depts:  system:dept:{list,add,edit,remove}
```

判定 `AuthenticatedUser.has_permission()` = `is_superuser or required in permissions`。权限不入 JWT，每请求从 DB 聚合（`User→UserRole→Role→RoleMenu→Menu`，[`src/auth/repository.py`](../src/auth/repository.py)）。
**额外守卫**：`_require_assignable_roles` 防提权（授出的权限须为 actor 权限子集）。

### 4.5 🟡 数据权限（data_scope 6 档）：判定全实现，已应用 user + role 域

**✅ 判定全实现**（[`src/auth/service.py`](../src/auth/service.py) `resolve_data_scope`）：`all / custom / dept / dept_and_child / self / dept_and_child_or_self` 六档齐全；多角色取并集、最宽优先；`dept_and_child` 走邻接表子树遍历（`_collect_subtree`）；部门树缓存 + 变更失效（[`src/auth/dept_tree_cache.py`](../src/auth/dept_tree_cache.py)）；无部门时 fail-closed。

**✅ 已应用**：
- user 域：[`src/admin/users/repository.py`](../src/admin/users/repository.py) 按 `User.department_id` + `User.id` 翻成 SQL WHERE，并用于列表 + 单条 mutation 可见性守卫。
- role 域：[`src/admin/roles/repository.py`](../src/admin/roles/repository.py) 按 `Role.create_dept` + `Role.created_by` 翻成 SQL WHERE，并用于列表 + 单条 mutation 可见性守卫。

**⚠️ 缺口**：
- **department 列表不过滤**（树导航的结构性需要，待定）。
- **其他域全无**：api_key / grant / quota / model_catalog / channel。
- **M3 新业务域要沿用 `create_dept`/`created_by` 通用规则**，禁止再按端点临时拼权限。

### 4.6 ✅ 超级管理员 bypass：两条线均覆盖

`_SUPERUSER_PERM = "*:*:*"` 哨兵（`src/auth/service.py`）。验证覆盖：
- **接口权限**：`has_permission` → `is_superuser` 短路；`RequiredPerms` 依赖随之放行。
- **数据权限**：`resolve_data_scope` 检测 `is_superuser` → 直接返回 `unrestricted=True`。
- **提权守卫**：`UserService` 中 `actor.is_superuser` 跳过 `_require_assignable_roles`。

**说明**：无 `is_admin`/`platform_admin` 命名标记，纯靠 `*:*:*` 通配（架构决策：否决段级通配，分组授权由菜单父节点表达）。这是有意设计，非缺口。

### M1：用户 / 凭证 / RBAC 🟡

**📁 已落地**：
- **核心安全层** [`src/core/security.py`](../src/core/security.py)：argon2 密码哈希（verify 不匹配/畸形均折叠 False）、sk-key sha256 校验、平台自签 JWT（**硬锁 RS256**，decode 显式传 `algorithms=["RS256"]` 防算法混淆，claims `extra="forbid"`，最小集 `sub/iat/exp/token_use`，权限不入 JWT，`sub=sys_user.id`）。
- **auth 域** `src/auth/`：login + me 端点；`AuthRepository`（权限码聚合）；`AuthService`（账密校验 / token 与 key 解析 / RBAC 判定 / 数据范围解析，frozen DTO `AuthenticatedUser` + `DataScopeFilter`）；`credentials.py`（统一 HTTP 凭据识别）；`middleware.py`（`AuthMiddleware` + `PUBLIC_PATHS` + `sk-key` 协议端点白名单）；`dependencies.py`（`CurrentUser` 给 LLM/MCP 协议处理器，`CurrentJwtUser` 给管理面，`TraceId` / `RequiredPerms`）；认证失败折叠不透明码（坏凭据/坏 token/禁用用户 → `auth.invalid_token` 401，越权 → `auth.forbidden` 403）。
- **用户/角色/部门 CRUD** `src/admin/`（提前完成 M3 一部分）：departments（环路守卫 + 删除阻断）；roles（RoleMenu/RoleDept 链 + code 唯一 + 删除清 UserRole）；users（UserRole 分配 + argon2 + 3 道安全守卫：越权创建/移动防护、角色分配提权防护、工号唯一）。挂 `/admin`，perm 门禁 `system:{dept,role,user}:*`。
- **数据权限基础**：`BaseEntity.create_dept` 已补；user 域按 `department_id`，role 域按 `create_dept/created_by`。
- **凭据边界**：sk-key 只允许 LLM 推理协议端点与 MCP 单协议端点，不再等同后台 JWT；后台管理面与资源管理 CRUD 均 JWT-only。

**⚠️ 缺口**（关键，决定 M1 不能算纯 ✅）：
1. **登录生命周期不全** → 缺 logout/refresh/改密/个人中心（见 4.1）。
2. **菜单 CRUD 端点未写** → perms 码目前靠迁移/手工种。
3. **M3 业务域数据权限待随端点落地** → credentials/catalog/grants/usage/quota 需沿用 `create_dept/created_by` 通用规则。

**✅ 质量门**：当前测试套件 **230 passed**，`mypy --strict` 0 error、ruff clean。

## 七、全局约束（贯穿所有里程碑）

- **类型安全零容忍**：禁 `# type: ignore`，`mypy --strict` 是 CI 门禁。
- **测试是收工前置**：新功能/修 bug 必带测试（红 → 绿）；禁删/skip 失败用例骗 CI；单测不碰共享 PG/Redis（用内存 SQLite + 纯异步 Redis double）。
- **四层单向依赖**：router → service → repository → db；router 薄、service 厚。
- **窄腰 + 最小依赖**：新依赖必带版本上界；新基建第一个真实消费者出现再引入（无消费者的基建是过度工程）。
- **prompt 缓存神圣**（网关侧）：不中途改上下文/换工具集/重建系统提示。
- **i18n 纯 code**：后端只发 code，前端拥有展示文本；枚举 label 后端单源 + codegen 同步；不建 DB 字典表。
- **配置/密钥分离**：`.env` 仅放密钥；行为配置走 `config.py`。
- **关联无物理外键**：存 id + 普通索引，引用完整性由 service 维护（数据模型 0.7）。
