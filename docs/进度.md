## 〇、系统全景（30 秒回顾，细节见架构稿）

公司统一内部 AI 基础设施平台，一个模块化单体（FastAPI），纳管两类 AI 能力：

```
客户端（Hermes / Claude Code / OA / 脚本）
   │ 同一套身份：企微 JWT  或  sk-xxxx
   ▼
统一平台（src/ 一个 repo）
   ├─ auth/     认证 + RBAC + 数据权限           （地基，M1）
   ├─ admin/    管理后台 API（用户/角色/部门/号池/模型/配额）  （M1 部分 + M3）
   ├─ gateway/  LLM 网关：三协议原生直通 + litellm.Router  （M2，✅ 完成）
   ├─ mcp/      标准 MCP 服务器（FastMCP，零新表）        （M4，未动工）
   └─ tasks/    ARQ 后台任务（用量汇总/配额重置/清理）     （M5，骨架存在）
   共享：PostgreSQL + Redis
```

身份模型（架构稿 G5）：**主体落库统一是 `sys_user.id`（代码/API 命名 `user_id`）**；凭据分层：管理后台与资源管理 API 只接受平台 JWT/企微 JWT；LLM 推理协议端点与 MCP 协议端点额外接受 `sk-key` 作为程序调用凭据。二者最终都归因到同一个 `user_id` 与配额桶，但 `sk-key` 不是后台登录态。

- **M1 先于一切业务面**：网关与后台都依赖鉴权地基；先做地基，否则各自重造一套鉴权。

- **M4 晚于 M2/M3**：MCP 经服务层暴露能力，须等服务边界稳定。
- **M6 最后做**：企微是外部适配器，用户/RBAC 用平台自签凭证已可独立闭环、完全可测。

### 4.6 ✅ 超级管理员 bypass：两条线均覆盖

超管判定 = **持有 `code == "superadmin"` 的在用角色**（`AuthenticatedUser.is_superuser` → `SUPERADMIN_ROLE_CODE in role_codes`，常量在 [`src/auth/constants.py`](../src/auth/constants.py)）。验证覆盖：
- **接口权限**：`has_permission` → `is_superuser` 短路；`RequiredPerms` 依赖随之放行。
- **数据权限**：`resolve_data_scope` 检测 `is_superuser` → 直接返回 `unrestricted=True`。
- **提权守卫**：`UserService` 中 `actor.is_superuser` 跳过 `_require_assignable_roles`；且把 `superadmin` 角色本身列为非超管不可授出（防经角色分配间接提权）。

**说明**：超管由**角色身份**（`superadmin` role code）承载，而非某个魔法权限字符串——bootstrap-safe（不依赖 menu/role_menu 链，种子管理员建出来即生效）。早期文档曾写「靠 `*:*:*` 通配权限」，与代码不符，已据实更正。`has_permission` 仍支持精确权限码匹配，但「全局放行」走的是 role code 这条线，不是通配权限。

### 6.1 刻意推迟项（非地基缺口，记录理由防遗忘）

以下经 Oracle 审计识别、但**刻意不在本轮落地**，因其违反「无消费者的基建是过度工程」或属边缘层职责，不阻塞业务开工：

- **指标监控（Prometheus / OpenTelemetry）**：是横切观测基建。M2 网关热路径已落地，指标接入时机已到——随网关**真实上量部署**时统一接入（DB/Redis pool、接口耗时、登录失败、上游调用等指标在有真实流量时才有意义）。现已有结构化日志 + trace_id + 健康探针兜底可观测性。
- **通用接口限流（登录外）**：架构定位 nginx 为反代 + 负载均衡层，通用限流是**边缘层职责**，应在 nginx（`limit_req`）落地，不在应用层重造一套。登录维度限流（按用户名锁定 + 按 IP 滑窗）因涉及业务语义（防爆破 + 防枚举）已在应用层实现。
- **登录日志写放大**：登录被锁定后仍写 login_log 留痕（安全审计需要）。2000 员工规模下写量可接受；随 **M2** 整体用量上量后再评估「锁定后降采样 / 审计异步化」。
- **JWT `iss` / `aud` claims**：仅在「多服务共享同一签名密钥」场景才防 token confusion；本平台 JWT 仅自签自用，且 RS256 已硬锁（`decode` 显式传 `algorithms=["RS256"]`）防住真正的算法混淆攻击面。加 iss/aud 是低收益的契约变更，留待真有多服务消费者时再做。

## 七、全局约束（贯穿所有里程碑）

- **类型安全零容忍**：禁 `# type: ignore`，`mypy --strict` 是 CI 门禁。
- **测试是收工前置**：新功能/修 bug 必带测试（红 → 绿）；禁删/skip 失败用例骗 CI；单测不碰共享 PG/Redis（用内存 SQLite + 纯异步 Redis double）。
- **四层单向依赖**：router → service → repository → db；router 薄、service 厚。
- **窄腰 + 最小依赖**：新依赖必带版本上界；新基建第一个真实消费者出现再引入（无消费者的基建是过度工程）。
- **prompt 缓存神圣**（网关侧）：不中途改上下文/换工具集/重建系统提示。
- **i18n 纯 code**：后端只发 code，前端拥有展示文本；枚举 label 后端单源 + codegen 同步；不建 DB 字典表。
- **配置/密钥分离**：`.env` 仅放密钥；行为配置走 `config.py`。
- **关联无物理外键**：存 id + 普通索引，引用完整性由 service 维护（数据模型 0.7）。
