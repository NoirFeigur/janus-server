## 〇、系统全景（30 秒回顾，细节见架构稿）

公司统一内部 AI 基础设施平台，一个模块化单体（FastAPI），纳管两类 AI 能力：

```
客户端（Hermes / Claude Code / OA / 脚本）
   │ 同一套身份：企微 JWT  或  sk-xxxx
   ▼
统一平台（src/ 一个 repo）
   ├─ auth/     认证 + RBAC + 数据权限           （地基，M1）
   ├─ admin/    管理后台 API（用户/角色/部门/号池/模型/配额）  （M1 部分 + M3）
   ├─ gateway/  LLM 网关：三协议入口 + litellm.Router  （M2，未动工）
   ├─ mcp/      标准 MCP 服务器（FastMCP，零新表）        （M4，未动工）
   └─ tasks/    ARQ 后台任务（用量汇总/配额重置/清理）     （M5，骨架存在）
   共享：PostgreSQL + Redis
```

身份模型（架构稿 G5）：**主体只有「账户」一种**；凭据有 **企微 JWT** + **sk-key** 两种，解析到同一账户、同一配额桶。

**排序铁律**（违反会返工）：
- **M1 先于一切业务面**：网关与后台都依赖鉴权地基；先做地基，否则各自重造一套鉴权。
- **M2 配额原语就地建**：Redis 配额计数/失效原语随 M2 消费者落地，不在 M0 提前堆砌。
- **M3 依赖 M1 的 RBAC**：后台 CRUD 的权限门禁必须建立在 M1 判定之上。
- **M4 晚于 M2/M3**：MCP 经服务层暴露能力，须等服务边界稳定。
- **M6 最后做**：企微是外部适配器，账户/RBAC 用平台自签凭证已可独立闭环、完全可测。

## 二、后台管理系统基本盘清单（直面「是否具备后台基本要求」）

> 以成熟企业级后台（若依 RuoYi 等）为标尺，逐项核对**当前代码**是否具备。这是衡量「能不能算一个后台」的硬指标。

| 能力项 | 状态 | 说明 / 缺口 |
|---|---|---|
| 账密登录 | ✅ | `POST /auth/login`，argon2 校验，发 RS256 自签 JWT |
| 当前用户信息 | ✅ | `GET /auth/me`，返回账户 + 权限码集合 |
| 退出登录（登出） | ⚠️ 缺 | 无登出端点、无令牌吊销/黑名单（当前 JWT 无状态、不可撤销） |

| 个人中心（改资料/头像/偏好语言） | ⚠️ 缺 | 无端点 |
| **统一登录拦截** | ❌ 缺 | **无 auth 中间件**，逐端点 `Depends` 散落，漏挂即裸奔（详见第四章） |

| 角色管理 CRUD | ✅ | `src/admin/roles/`，含 RoleMenu/RoleDept 链管理 |
| 部门管理 CRUD | ✅ | `src/admin/departments/`，含环路守卫 + 删除阻断 |
| 菜单/权限（perms）管理 | ⚠️ 部分 | `sys_menu` 表 + perms 码鉴权链路就绪；**菜单 CRUD 端点未写**（perms 目前靠迁移/手工种） |
| 接口权限（perms 码门禁） | ✅ | 12 个 admin 端点全挂 `RequiredPerms`（详见第四章） |
| 数据权限（data_scope 6 档） | 🟡 | 判定全实现，但**仅 user 域真正过滤**；role/dept/其他域未应用（详见第四章） |
| 超级管理员无视权限 | ✅ | `*:*:*` 短路接口权限 + 数据权限**两条线**（详见第四章） |
| 分页 / 排序 | ❌ 缺 | 列表端点尚无统一分页；`src/core/pagination.py` 未建（随首个分页端点落地） |
| 操作日志（谁改了什么） | ❌ 缺 | 未建 `sys_admin_audit_log`；设计已声明「将来合规要求时再统一加表」 |
| 登录日志 | ❌ 缺 | 未记录登录历史/IP |
| 在线用户 / 令牌管理 | ❌ 缺 | 无状态 JWT 下需额外令牌台账，未做 |

**结论**：用户/角色/部门三大件 CRUD + RBAC 判定已具备，**但缺三块后台硬骨头**——
① 统一鉴权拦截 + 开放白名单；② 登录生命周期补全（登出/刷新/改密/个人中心）；③ 操作审计/分页等运营基础。
这三块是「具备后台基本要求」的关键缺口，已登记进第六章待办册。

## 三、数据模型现状与漂移（对齐数据模型设计 v0.5.5）

### 3.1 已落地 16 张表

`src/db/models/` 按领域拆分，全部继承三类基类之一：

| 文件 | 模型（类名 → 表名） |
|---|---|
| `identity.py` | User→`sys_user`、Department→`sys_department`、UserOAuth→`sys_user_oauth`、Role→`sys_role`、Menu→`sys_menu`、UserRole→`sys_user_role`、RoleMenu→`sys_role_menu`、RoleDept→`sys_role_dept` |
| `credential.py` | ApiKey→`sys_api_key` |
| `model_catalog.py` | UpstreamChannel→`sys_upstream_channel`、ChannelKey→`sys_channel_key`、LogicalModel→`sys_logical_model`、ModelDeployment→`sys_model_deployment` |
| `grant.py` | UserModelGrant→`sys_user_model_grant` |
| `usage.py` | UsageRecord→`sys_usage_record`（`LogEntity`） |
| `quota.py` | Quota→`sys_quota` |

三类基类（[`src/db/base.py`](../src/db/base.py)）：
- `BaseEntity`：`id / is_deleted / created_by / created_at / updated_by / updated_at`（12 张配置/业务表继承）
- `LogEntity`：`id / created_at`（append-only 流水，UsageRecord）
- `LinkEntity`：`id / created_at`（关联表，物理删）

### 3.2 ⚠️ 缺口：`BaseEntity` 缺 `create_dept`（部门级数据权限的硬前提）

**现状**：`BaseEntity` 没有 `create_dept` 列；数据模型文档 v0.5.5 的 `BaseEntity` 定义**同样没有** → **代码 + 设计文档双缺口**。

**为什么是真缺口**：数据权限 6 档里，`self` 档靠 `created_by` 过滤、`dept`/`dept_and_child` 档需要「这行数据归属哪个部门」。
当前 user 域的数据过滤走的是 `User.department_id`（用户**自己**所在部门），这对「用户管理」屏恰好成立；
但**任何通用业务表**（quota / grant / channel_key / api_key …）都没有部门归属列，部门级数据权限对它们**无从过滤**。
这正是若依在业务表统一加 `create_dept`（= 创建者当时所在部门）的原因。

**待定的设计决策**（实现前必须拍）：
1. `create_dept` 加在 `BaseEntity`（覆盖全部 12 张业务表），insert 时由 service 从 actor 的部门写入。
2. user 域数据权限按 `department_id`（用户自身部门）还是 `create_dept`（记录创建部门）过滤——**二者语义不同**，需明确（若依对 user 用 `dept_id`，对其他业务表用 `create_dept`）。
3. 数据模型文档 `BaseEntity` 同步补列 + 修订版本号。

### 3.3 ⚠️ 漂移：表名 `sys_` 前缀

代码给**所有**表加了 `sys_` 前缀（`sys_api_key` / `sys_usage_record` / `sys_quota` / `sys_upstream_channel` …）；
数据模型文档第 2/3 批表名（`api_key` / `usage_record` / `quota` / `upstream_channel` …）**未带前缀**。
代码更一致（全 `sys_`），**待同步**到数据模型文档。

## 四、鉴权与权限体系现状（全链路核实）

> 这是用户重点追问的部分：登录接口、统一拦截、开放白名单、接口权限、数据权限、超管 bypass。逐条给代码现状 + 缺口。

### 4.1 登录接口

**✅ 已落地**（[`src/auth/router.py`](../src/auth/router.py)）：

| 端点 | 方法 | 说明 |
|---|---|---|
| `/auth/login` | POST | 账密 → RS256 自签 access token |
| `/auth/me` | GET | 当前账户 + 权限码集合（需有效凭据） |

**服务层解析器**（无 HTTP 面，[`src/auth/service.py`](../src/auth/service.py)）：
`resolve_access_token`（JWT 验签）、`resolve_api_key`（sk-key → sha256 → 查 `sys_api_key`）、`authenticate_password`（发 JWT）。

**⚠️ 缺口**：refresh（恒 None，留 M6）、logout（无吊销机制）、改密、个人中心、企微回调（`OAuthSource.wecom` 枚举 + `UserOAuth` 模型存在，但无端点无逻辑，留 M6）。

### 4.2 ❌ 统一登录拦截：不存在（当前散落）

[`src/main.py`](../src/main.py) 仅挂两个中间件：`TraceIdMiddleware`（trace-id）、`LocaleMiddleware`（i18n）——**都不做鉴权**。
`include_router` **无 `dependencies=[...]`**。鉴权完全靠**逐端点** `Depends`：

```python
# auth 域：显式 CurrentAccount
async def me(account: CurrentAccount, ...)
# admin 域：RequiredPerms（内部依赖 get_current_account）
account: Annotated[..., Depends(RequiredPerms("system:user:list"))]
```

**风险**：每加一个端点都要手动挂鉴权，**漏挂即裸奔**，无中央兜底。
**目标**（登记待办）：加 `AuthMiddleware` 复用 `_extract_credential` 解析凭据 → 写 `request.state.account`，配 `PUBLIC_PATHS` 放行；`RequiredPerms` 保留做 RBAC 门禁。把鉴权从「散落」收敛为「统一」。

### 4.3 ❌ 开放接口白名单：不存在（隐式开放）

全 `src/` 搜 `PUBLIC` / `whitelist` / `open_paths` / `skip_auth` **零命中**。当前「开放」靠端点**没挂依赖**隐式成立：
`/auth/login`、`/health/live`、`/health/ready`、`/docs`、`/openapi.json`、空壳 `gateway` 路由。
**目标**：定义中央 `PUBLIC_PATHS` frozenset，与统一拦截配套，让「哪些免鉴权」成为显式、可审计的清单。

### 4.4 ✅ 接口权限（perms 码）：admin 域完善

`RequiredPerms("域:资源:动作")` 依赖工厂（[`src/auth/dependencies.py`](../src/auth/dependencies.py)），挂在全部 12 个 admin 端点：

```
users:  system:user:{list,add,edit,remove}
roles:  system:role:{list,add,edit,remove}
depts:  system:dept:{list,add,edit,remove}
```

判定 `AuthenticatedAccount.has_permission()` = `is_superuser or required in permissions`。权限不入 JWT，每请求从 DB 聚合（`User→UserRole→Role→RoleMenu→Menu`，[`src/auth/repository.py`](../src/auth/repository.py)）。
**额外守卫**：`_require_assignable_roles` 防提权（授出的权限须为 actor 权限子集）。

### 4.5 🟡 数据权限（data_scope 6 档）：判定全实现，应用仅 user 域

**✅ 判定全实现**（[`src/auth/service.py`](../src/auth/service.py) `resolve_data_scope`）：`all / custom / dept / dept_and_child / self / dept_and_child_or_self` 六档齐全；多角色取并集、最宽优先；`dept_and_child` 走邻接表子树遍历（`_collect_subtree`）；部门树缓存 + 变更失效（[`src/auth/dept_tree_cache.py`](../src/auth/dept_tree_cache.py)）；无部门时 fail-closed。

**🟡 应用仅 user 域**：仅 [`src/admin/users/repository.py`](../src/admin/users/repository.py) 的 `_scope_predicate` 真正把 scope 翻成 SQL WHERE（过滤 `User.department_id`），并用于 `list_in_scope` + 每次 mutation 的可见性守卫。

**⚠️ 缺口**：
- **role 列表不过滤**：`RoleService.list_roles` 无 data_scope，任何有 `system:role:list` 的人看全部角色。
- **department 列表不过滤**（树导航的结构性需要，待定）。
- **其他域全无**：api_key / grant / quota / model_catalog / channel。
- **依赖 3.2 的 `create_dept`**：通用业务表要做部门级过滤，必须先补 `create_dept`。

### 4.6 ✅ 超级管理员 bypass：两条线均覆盖

`_SUPERUSER_PERM = "*:*:*"` 哨兵（`src/auth/service.py`）。验证覆盖：
- **接口权限**：`has_permission` → `is_superuser` 短路；`RequiredPerms` 依赖随之放行。
- **数据权限**：`resolve_data_scope` 检测 `is_superuser` → 直接返回 `unrestricted=True`。
- **提权守卫**：`UserService` 中 `actor.is_superuser` 跳过 `_require_assignable_roles`。

**说明**：无 `is_admin`/`platform_admin` 命名标记，纯靠 `*:*:*` 通配（架构决策：否决段级通配，分组授权由菜单父节点表达）。这是有意设计，非缺口。

**⚠️ 缺口/遗留**：Pydantic 422 校验错误 `errors[].msg` 仍发英文原文（i18n 方向待定）。

### M1：账户 / 凭证 / RBAC 🟡

**📁 已落地**：
- **核心安全层** [`src/core/security.py`](../src/core/security.py)：argon2 密码哈希（verify 不匹配/畸形均折叠 False）、sk-key sha256 校验、平台自签 JWT（**硬锁 RS256**，decode 显式传 `algorithms=["RS256"]` 防算法混淆，claims `extra="forbid"`，最小集 `sub/iat/exp/token_use`，权限不入 JWT）。
- **auth 域** `src/auth/`：login + me 端点；`AuthRepository`（权限码聚合）；`AuthService`（账密校验 / token 与 key 解析 / RBAC 判定 / 数据范围解析，frozen DTO `AuthenticatedAccount` + `DataScopeFilter`）；`dependencies.py`（`CurrentAccount` / `TraceId` / `RequiredPerms`，凭据解析 `X-API-Key` 优先 → Bearer）；认证失败折叠不透明码（坏凭据/坏 token/禁用账户 → `auth.invalid_token` 401，越权 → `auth.forbidden` 403）。
- **用户/角色/部门 CRUD** `src/admin/`（提前完成 M3 一部分）：departments（环路守卫 + 删除阻断）；roles（RoleMenu/RoleDept 链 + code 唯一 + 删除清 UserRole）；users（UserRole 分配 + argon2 + 3 道安全守卫：越权创建/移动防护、角色分配提权防护、工号唯一）。挂 `/admin`，perm 门禁 `system:{dept,role,user}:*`。

**⚠️ 缺口**（关键，决定 M1 不能算纯 ✅）：
1. **`BaseEntity.create_dept` 缺失** → 部门级数据权限对通用业务表无从落地（见 3.2）。
2. **数据权限仅 user 域应用** → role/其他域未过滤（见 4.5）。
3. **统一登录拦截缺失** → 逐端点散落（见 4.2）。

## 七、全局约束（贯穿所有里程碑）

- **类型安全零容忍**：禁 `# type: ignore`，`mypy --strict` 是 CI 门禁。
- **测试是收工前置**：新功能/修 bug 必带测试（红 → 绿）；禁删/skip 失败用例骗 CI；单测不碰共享 PG/Redis（用内存 SQLite + 纯异步 Redis double）。
- **四层单向依赖**：router → service → repository → db；router 薄、service 厚。
- **窄腰 + 最小依赖**：新依赖必带版本上界；新基建第一个真实消费者出现再引入（无消费者的基建是过度工程）。
- **prompt 缓存神圣**（网关侧）：不中途改上下文/换工具集/重建系统提示。
- **i18n 纯 code**：后端只发 code，前端拥有展示文本；枚举 label 后端单源 + codegen 同步；不建 DB 字典表。
- **配置/密钥分离**：`.env` 仅放密钥；行为配置走 `config.py`。
- **关联无物理外键**：存 id + 普通索引，引用完整性由 service 维护（数据模型 0.7）。
